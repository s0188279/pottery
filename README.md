# 🏺 Мобильное приложение гончарной мастерской «Глина»

Добро пожаловать в репозиторий клиентского мобильного приложения для гончарной мастерской «Глина». Проект реализован в рамках выполнения выпускного задания Летней школы.

Цель проекта — автоматизация процесса записи клиентов на мастер-классы, исключение овербукинга (двойных броней) и снижение операционной нагрузки на владельца студии.

---

## 🛠 Технологический стек и запуск

* **Фронтенд:** Single Page Application (SPA) на базе чистых веб-технологий: **HTML5, JavaScript (ES6+), CSS3**.
* **UI-фреймворк:** **Tailwind CSS** (через CDN) для адаптивной верстки под мобильные экраны.
* **Архитектура:** Имитация клиент-серверного взаимодействия (мокирование API контрактов на основе системного анализа предметной области).

### 🚀 Как запуск приложения:
1. Скачайте или клонируйте данный репозиторий.
2. Найдите файл `index.html` в корневой папке.
3. Дважды кликните по файлу `index.html`, чтобы открыть его в любом современном браузере.
4. *Рекомендуется:* Для лучшего пользовательского опыта откройте панель разработчика (клавиша `F12`), переключитесь в режим эмуляции мобильного устройства (Toggle Device Toolbar) и выберите любой смартфон (например, iPhone 12 Pro).

## 🗂 Структура проекта и артефакты

Проект разбит на логические этапы, согласно фреймворкам, пройденным на лекциях школы:

### 1. Этап анализа и проектирования (Системный анализ)
В этих файлах зафиксированы требования к MVP и архитектурные контракты:
* `business.md` — Бизнес-требования и цели проекта (БЦ-1, БЦ-2 и т.д.).
* `functional.md` — Функциональные требования по модулям (авторизация, расписание, бронирование).
* `non-functional.md` — Нефункциональные требования (UX, производительность, ограничения).
* `user-stories.md` — Пользовательские истории (User Stories) с критериями приемки (AC).
* `use-cases.md` — Подробные сценарии использования (Use Cases) с альтернативными путями.
* `domain-description.md` — Глоссарий и описание сущностей предметной области (`Client`, `Slot`, `Booking`).
* `data-model.md` — Логическая схема данных, статусы броней и ключевые бизнес-инварианты.
* `api-sequence.md` — Сценарии сетевого взаимодействия (Mermaid диаграммы) в критических точках.
* `customer-questions.md` — Перечень уточняющих вопросов к заказчику для устранения «серых зон».

### 2. Этап разработки (Реализованные фичи)
В рамках MVP клиентского приложения успешно спроектированы и закодированы 3 ключевые фичи (подробные промпты к ИИ и требования находятся в описаниях фич):
* `feature-1-auth.md` — **Фича 1:** Авторизация по номеру телефона (симуляция ввода OTP).
* `feature-2-schedule.md` — **Фича 2:** Просмотр сетки доступных занятий на 7 дней вперед.
* `feature-3-booking.md` — **Фича 3:** Оформление бронирования слота с динамическим выбором проката инвентаря.

### 3. Этап тестирования и стабилизации (Контроль качества)
В ходе ручного тестирования разработанного интерфейса были обнаружены, задокументированы и исправлены следующие критические дефекты:
* `bug-1.md` — **Баг 1:** Отсутствие валидации номера телефона (исправлено добавлением проверок длины строки).
* `bug-2.md` — **Баг 2:** Нарушение бизнес-инварианта овербукинга (исправлено динамическим вычитанием свободных мест `free_seats` при успешной брони).
