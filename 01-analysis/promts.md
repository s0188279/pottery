# 📝 История промптов ИИ для этапов анализа и проектирования

В данном документе зафиксированы переформулированные профессиональные промпты, которые использовались для генерации аналитических артефактов и архитектурных схем MVP мобильного приложения гончарной мастерской «Глина».

---

## 🧭 Этап 1. Анализ и выявление требований

### 1.1. Промпт для выявления серых зон и словаря предметной области
**Целевые файлы:** `customer-questions.md`, `domain-description.md`
**Входные данные:** `brief-pottery.md`

> **Текст промпта:**
> "Выступи в роли ведущего системного аналитика. На основе сырого брифа от заказчика Марины (владелицы гончарной мастерской «Глина», файл brief-pottery.md) необходимо провести первичный анализ требований и сформировать два документа в формате markdown:
> 
> 1. **customer-questions.md** — структурированный список уточняющих вопросов к заказчику для закрытия «серых зон». Сгруппируй вопросы по бизнес-логике (например, уточнение лимитов мест для продвинутых групп, если для новичков лимит 6 человек при 10 кругах), правилам и ограничениям отмены (предложи жесткий лимит отмены за 2 часа до начала вместо абстрактных '10 минут') и механике проката инвентаря.
> 2. **domain-description.md** — описание предметной области. Добавь глоссарий ключевых терминов (Слот расписания, Бронирование, Программа мастер-класса) и детальный атрибутивный состав основных бизнес-сущностей (Client, Program, Master, Slot, Booking) с указанием типов данных (UUID, String, DateTime, Enum, Decimal) для проектирования схемы."

---

### 1.2. Промпт для генерации пакета спецификаций (Бизнес, Функциональные и Нефункциональные требования)
**Целевые файлы:** `business.md`, `functional.md`, `non-functional.md`

> **Текст промпта:**
> "На основе ранее определенных сущностей и бизнес-контекста гончарной мастерской сформируй три структурированных технических документа в формате markdown:
> 
> 1. **business.md (Бизнес-требования):** Опиши текущие проблемы бизнеса (овербукинг, финансовые потери от поздних отмен, операционная загрузка в выходные) и переведи их в измеримые бизнес-цели (БЦ-1, БЦ-2...). Четко очерти границы проекта (In Scope / Out of Scope): в рамках MVP разрабатывается только мобильный интерфейс для роли 'Клиент' и контракты API для интеграции с существующим бэкендом. Онлайн-оплата и админ-панель владельца выносятся за рамки текущей поставки.
> 2. **functional.md (Функциональные требования):** Составь атомарный, пронумерованный список требований (FR-001, FR-002...). Включи модули авторизации по номеру телефона (симуляция OTP), просмотра сетки слотов на 7 дней, оформления записи с флагом проката инвентаря и фартука, а также управление бронированиями с блокировкой кнопки отмены менее чем за 2 часа до начала занятия.
> 3. **non-functional.md (Нефункциональные требования):** Зафиксируй технические и качественные ограничения (NFR-001, NFR-002...). Опиши требования к платформе (оптимизация под мобильные UI), эргономике (размеры кликабельных зон) и интеграции с бэкендом как с Black-Box источником истины, включая обязательную обработку сетевых таймаутов и ошибок 409 Conflict со стороны сервера."

---

### 1.3. Промпт для моделирования сценариев (User Stories & Use Cases)
**Целевые файлы:** `user-stories.md`, `use-cases.md`

> **Текст промпта:**
> "Необходимо декомпозировать функциональные требования на сценарии взаимодействия. Сгенерируй два markdown-документа:
> 
> 1. **user-stories.md (Пользовательские истории):** Напиши User Stories по классическому шаблону 'Я как [Роль] хочу [Действие], чтобы [Ценность]' для эпиков: Авторизация, Расписание (с фильтрацией по мастерам/программам), Бронирование с опцией проката и Отмена записи. Для каждой истории пропиши строгие, проверяемые критерии приемки (Acceptance Criteria).
> 2. **use-cases.md (Сценарии использования):** Детально опиши два ключевых Use Case по шагам (Основной успешный путь взаимодействия пользователя и системы). UC-1: Запись на мастер-класс с прокатом инвентаря (включи альтернативный сценарий конкурентной записи, когда места закончились на бэкенде в момент оформления). UC-2: Отмена бронирования клиентом (включи альтернативный сценарий попытки отмены менее чем за 2 часа до начала с блокировкой UI)."

## 🏗 Этап 2. Архитектура и проектирование

### 2.1. Промпт для проектирования ресурсной модели данных и инвариантов
**Целевой файл:** `data-model.md`

> **Текст промпта:**
> "Выступи в роли ведущего backend-архитектора. Спроектируй ресурсную модель данных и спецификацию сущностей для API клиентского мобильного приложения гончарной мастерской 'Глина'. Бэкенд является единственным источником истины, ресурсы Program, Master и Slot доступны клиенту в режиме Read-Only. 
> 
> Подготовь технический документ в markdown, содержащий:
> 1. Табличное описание атрибутивного состава ресурсов `Client` (с флагом `is_loyal`), `Program`, `Master`, `Slot`, `Booking` с указанием типов данных, ключей (PK/FK) и описанием полей.
> 2. Жизненный цикл сущности `Booking` в виде Mermaid-диаграммы статусов (`created`, `cancelled_by_client`, `cancelled_by_studio`, `completed`) и таблицу переходов.
> 3. Ключевые бизнес-инварианты и правила целостности: формулу динамического расчета свободных мест (`Slot.free_seats`), жесткую валидацию вместимости групп в зависимости от типа программы (`LEPKA` <= 6 мест, `KRUG` <= 10 мест) и допущения по фонду проката инвентаря для MVP."

---

### 2.2. Промпт для моделирования сетевого взаимодействия (Sequence Diagrams)
**Целевой файл:** `api-sequence.md`

> **Текст промпта:**
> "Разработай архитектурную спецификацию сетевого взаимодействия мобильного приложения и API-сервера в критических сценариях. Документ должен быть оформлен в markdown и включать:
> 
> 1. Сквозные правила взаимодействия: логику передачи токенов авторизации (Bearer Token), обработку ошибок 401 Unauthorized, правила атомарности транзакций на бэкенде для исключения овербукинга и регламент обработки сетевых таймаутов (10 секунд).
> 2. Две подробные диаграммы последовательности (Sequence Diagrams) на синтаксисе Mermaid:
>    - **Сценарий 1:** Создание бронирования с флагом проката (`createBooking`). Учти ветки ответов: 201 Created, 409 Conflict (места закончились), 401 Unauthorized и Ошибка сети/5xx.
>    - **Сценарий 2:** Отмена бронирования клиентом (`cancelBooking`). Учти ветки ответов: 200 OK, 422 Unprocessable Entity (нарушено правило 2 часов), 409 Conflict (уже отменено администратором).
> 3. Пошаговые таблицы с описанием механики и логики обработки каждого шага на стороне мобильного приложения и API для обеих диаграмм."
