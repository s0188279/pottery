# Sequence-диаграммы API-взаимодействия

> Этап 2. Архитектура и проектирование. Механика сетевого взаимодействия мобильного приложения и API-сервера гончарной мастерской в критических сценариях.

## Сквозные правила взаимодействия
- **Авторизация:** Все эндпоинты, кроме отправки/проверки OTP, требуют передачи заголовка `Authorization: Bearer <token>`. При коде `401 Unauthorized` приложение сбрасывает сессию и переводит пользователя на экран входа.
- **Атомарность операций:** Бэкенд гарантирует транзакционность создания брони. Овербукинг (запись двух клиентов на одно последнее место) исключен на уровне БД сервера.
- **Таймауты:** Базовый таймаут для запросов составляет 10 секунд. При возникновении сетевых ошибок (Network Timeout, 5xx) транзакция на клиенте не завершается, отображается системный notification с возможностью повтора.

## Сценарий 1: Создание бронирования с прокатом (`createBooking`)

Пользователь выбирает слот, отмечает чекбокс проката инструментов и фартука и нажимает «Подтвердить запись».

sequenceDiagram
    autonumber
    actor User as Клиент (App)
    participant App as Мобильное приложение
    participant API as Бэкенд API

    User->>App: Нажимает «Подтвердить запись» (needs_rent: true)
    App->>App: Лоадер, блокировка экрана от повторных нажатий
    App->>API: POST /api/v1/bookings { "slot_id": "UUID", "needs_rent": true }
    
    alt Успешное создание (201 Created)
        API-->>App: 201 Created { booking_id, status: "created", free_seats: X }
        App->>User: Экран успеха (BS-002) + Детали брони
    else Места закончились в процессе оформления (409 Conflict)
        API-->>App: 409 Conflict { "code": "no_available_seats" }
        App->>User: Информационное модальное окно: «Места только что закончились»
    else Токен авторизации протух (401 Unauthorized)
        API-->>App: 401 Unauthorized
        App->>App: Сброс локальной сессии
        App->>User: Редирект на экран входа по OTP (SCR-001)
    else Ошибка сети / Сервер лежит (500 / Timeout)
        API-->>App: Network Error / 503 Service Unavailable
        App->>User: Снекбар: «Ошибка соединения. Попробуйте еще раз» (Экран доступен)
    end

    ### Пошаговое описание сценария создания

| Шаг | Направление | Механика и логика обработки |
| :-- | :-- | :-- |
| 1-2 | Клиент -> App | Пользователь инициирует бронь. Важно заблокировать UI (показать спиннер), чтобы избежать отправки дублирующих запросов. |
| 3 | App -> API | Отправляется мутирующий POST-запрос с идентификатором слота и флагом `needs_rent: true`. |
| 4-5 | API -> App (201) | Сервер в транзакции уменьшил количество мест. Приложение кэширует данные и переводит юзера на экран успеха. |
| 6-7 | API -> App (409) | Кейс конкурентной записи. Сервер вернул ошибку, приложение разблокирует интерфейс и возвращает пользователя на расписание с актуализацией мест. |

## Сценарий 2: Отмена бронирования клиентом (`cancelBooking`)

Пользователь нажимает «Отменить запись» на экране деталей бронирования.

sequenceDiagram
    autonumber
    actor User as Клиент (App)
    participant App as Мобильное приложение
    participant API as Бэкенд API

    User->>App: Нажимает «Отменить запись»
    App->>User: Показ модального окна подтверждения отмены
    User->>App: Подтверждает отмену
    App->>App: Блокировка UI, отображение лоадера
    App->>API: POST /api/v1/bookings/{id}/cancel
    
    alt Успешная своевременная отмена (200 OK)
        API-->>App: 200 OK { booking_id, status: "cancelled_by_client" }
        App->>User: Обновление статуса на экране деталей брони, скрытие кнопки отмены
    else До начала занятия осталось менее 2 часов (422 Unprocessable Entity)
        API-->>App: 422 Unprocessable Entity { "code": "late_cancellation_forbidden" }
        App->>User: Модалка: «Отмена невозможна. До начала мастер-класса осталось менее 2 часов»
    else Бронирование уже отменено администратором (409 Conflict)
        API-->>App: 409 Conflict { "code": "already_cancelled_by_studio" }
        App->>User: Уведомление: «Статус записи изменился», обновление экрана
    end

    ### Пошаговое описание сценария отмены

| Шаг | Направление | Механика и логика обработки |
| :-- | :-- | :-- |
| 1-4 | Клиент -> App | Двухэтапное подтверждение отмены UI для исключения случайных кликов. |
| 5 | App -> API | Запрос на отмену ресурса бронирования. Передается ID конкретной записи. |
| 6-7 | API -> App (200) | Сервер проверяет разницу во времени. Если > 2 часов, статус меняется на `cancelled_by_client`, место освобождается. Приложение обновляет карточку. |
| 8-9 | API -> App (422) | Сервер отклоняет запрос, так как нарушено бизнес-правило 2 часов. Приложение выводит строгое предупреждение, кнопка отмены в UI блокируется. |
